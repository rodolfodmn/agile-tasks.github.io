Bis2Crum

Método Scrum adaptado para Bis2bis.

Resumo

Um time composto por devs e um ou mais testers coringas ou não, entregam as features disponibilizadas pelo Gerente de Projeto, onde o time gerencia como as features vão ser desenvolvidas, dividindo as features em tasks, e podendo dividir as features em mais features caso necessário, sendo que uma feature, deve ser um entregável para o cliente. Cada feature possui as tasks obrigatórias: testes, correções, deploy e ajustes, sendo que os testes são realizados pelos testers e as correções e deploy pelos devs. Os testers devem saber se a feature está entregável ou não, sendo entregável, ela é deployda e disponibiliza para o Gerente de Projeto. Toda semana o time se reúne e valida as features que vão ser desenvolvidas durante a semana, sendo que se hover alguma task da semana passada que não foi entregue ela simplesmente é reanalisada. Caso haja algum bloqueio para algum dev, esse bloqueio é passado para o time, e caso o time não conseguir resolver, é passado para o Gerente de Projeto, sendo função dele resolver o bloqueio. Caso seja necessário inserir um novo desenvolvimento no que foi planejado para a semana, o desenvolvimento é pontuado e adicionado na fila imediata, sendo que será o próxima task imediata a ser realizada pelo time.

O Método

  • O Gerente de projeto cria o projeto na ferramenta de gerência, e alimenta o projeto com as features referenciadas com o documento de requisitos
  • Reunião de apresentação do projeto, onde o Gerente de projeto da uma visão geral do projeto para o time
  • Reunião do time para analise do entregável da semana, com a pontuação normal do scrum
  • Desenvolvimento da feature
    • o dev seleciona uma feature para inciar o desenvolvimento
    • o dev seleciona quebra a feature em tasks
    • com todas as tasks desenvolvidas, inicia o teste
      • o tester testa e insere os erros encontrados na task correções
      • o dev analisa e aplica as correções necessárias
      • o tester identificando que a feature está sem erros e funcionando como esperado tesmina a task
    • o dev da feature deploya a feature e passa para o Gerente
    • Caso exista alguma falta na feature o Gerente pode informar as mesmas na task faltas
    • o dev analisa e reinicia o dev da feature até o Gerente aprovar
    • Com a aprovação do Gerente, a task ajustes é concluída, concluindo o desenvolvimento da feature

Coisas boas do scrum atual

  • Kanban
    • Deixa tudo bem visível e de fácil entendimento
    • Ajuda o acompanhamento do Gerente
  • Sprint
    • A ideia de ter um período de entregáveis é bem legal
    • Ajuda a manter o ritmo em projetos grandes
    • Ajuda a mensurar o que é entregável
  • Pontuação poker
    • A ideia de ter a opinião de todos do time sobre o tempo de entrega é excelente
    • Cada integrate do time consegue ver pontos diferentes na feature
    • A estimativa fica muito mais real
  • Histórias e tasks
    • Ajudam na organização do projeto
    • A estimativa fica muito mais real, devido ao melhor entendimento da entrega

Coisas desnecessárias do scrum atual

  • Planning
    • Tira muito tempo de todos envolvidos no projeto
    • É muito corrida e sempre acaba ficando task mal feita, sendo que 1h ou pouco mais é insuficiente para quebrar tudo o que 3 ou mais devs vai fazer durante uma sprint!
    • Os devs podem muito bem fazer o planejamento do desenvolvimento da sua feature assim que inicia ela, com calma e precisão
    • Os deves podem contatar o Gerente sem problemas para tirar qualquer duvida da feature, caso seja complexo marque uma reunião rápida
  • Daily
    • O sistema já guarda o que foi feito no dia anterior
    • O que será feito durante o dia já está visível no sistema
    • São pausas que custam 15min de pausa, quebra do raciocínio que custa tanto na pausa como na retomada, julgando isso acredito que traz um atraso de 30 ~ 40min no dia de cada dev
  • Retrospectiva
    • O sistema já guarda o que foi feito no dia anterior, como na Daily
    • São pausas que custam 1h de pausa ou mais, quebra do raciocínio que custa tanto na pausa como na retomada, julgando isso acredito que traz um atraso de 1h30 ~ 1h40min de cada dev
    • Em alguns casos pode deixar toda a parte da tarde improdutiva custando 4h de cada dev.
  • Scrum Master
    • O Gerente de projeto pode muito bem colocar as tasks no sistema
    • Os devs podem gerenciar muito bem as tasks e ver o andamento do projeto
    • Os deves podem contatar o Gerente sem problemas e o Gerente os devs
